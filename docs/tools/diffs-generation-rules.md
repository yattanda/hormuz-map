# index_html_diffs.md 生成ルール

> **注意：このファイルは Claude.ai で `docs/tools/index_html_diffs.md` を生成・更新する際の参照ルールであり、Claude Code の通常作業では常時参照しない。**

---

## ⚡ モバイル更新対応：APPLY ブロック形式（必須）

GitHub Actions による自動適用を有効にするため、
**str_replace の old_str / new_str は必ず以下の APPLY ブロックで囲む**。

```html
<!-- APPLY:START -->
<!-- OLD:START -->
（ここに index.html から完全一致で抽出した置き換え前文字列）
<!-- OLD:END -->
<!-- NEW:START -->
（ここに置き換え後の文字列）
<!-- NEW:END -->
<!-- APPLY:END -->
```

### ルール

- `<!-- OLD:START -->` の直後・`<!-- OLD:END -->` の直前は改行のみ（余分なスペース不可）
- old_str は **index.html の実際の文字列と完全一致**させる（スペース・改行を含む）
- 1セクションに複数の str_replace が必要な場合は APPLY ブロックを複数並べる
- APPLY ブロック外のセクションヘッダー（`## [S01]` 等）や説明文は自由に書いてよい

### 例

```markdown
## [S01] ヘッダー日時更新

<!-- APPLY:START -->
<!-- OLD:START -->
📅 2026年5月22日 09:30 JST
<!-- OLD:END -->
<!-- NEW:START -->
📅 2026年5月23日 11:00 JST
<!-- NEW:END -->
<!-- APPLY:END -->

## [S02] TICKER更新

<!-- APPLY:START -->
<!-- OLD:START -->
<div id="ticker-text">古いTICKER文</div>
<!-- OLD:END -->
<!-- NEW:START -->
<div id="ticker-text">新しいTICKER文</div>
<!-- NEW:END -->
<!-- APPLY:END -->
```

### old_str の一意性ルール（必須）

- old_str は **最低100文字以上** を目安にする
- 短い文字列（日付・数字・短フレーズ）は必ず前後の固有HTMLタグを含めて一意にする

❌ NG例（短すぎ・複数箇所にマッチ）:
  2026年5月23日 09:58 JST

✅ OK例（前後のHTMLで一意化）:
  <span class="badge-item badge-date">📅2026年5月23日 09:58 JST</span>

❌ NG例:
  85日目

✅ OK例:
  <span class="countdown-days">85</span>日目

---

### ❌ APPLY ブロック内の禁止事項（必須）

以下は APPLY ブロック内に絶対に含めてはいけない：

1. **Markdownコードフェンス（` ``` `）**
   - OLD:START〜OLD:END の間は HTML をそのまま記載する
   - Markdown の ` ```html ` や ` ``` ` で囲まない
   - コードフェンスが混入すると old_str に ` ``` ` が含まれ
     index.html に存在しない文字列となり「見つからない」エラーになる

   ❌ NG:
   ```
   <!-- OLD:START -->
   ` ```html `
   <div>...</div>
   ` ``` `
   <!-- OLD:END -->
   ```

   ✅ OK:
   ```
   <!-- OLD:START -->
   <div>...</div>
   <!-- OLD:END -->
   ```

2. **短すぎる old_str（100文字未満は原則禁止）**
   - 日付文字列・短いフレーズ単独での old_str は複数箇所にマッチする
   - 必ず前後の固有 HTML タグ・属性・クラス名を含めて一意にする

   ❌ NG（短すぎ・複数マッチ）:
   ```
   <!-- OLD:START -->
   2026年5月24日 08:12 JST
   <!-- OLD:END -->
   ```

   ✅ OK（前後のHTMLで一意化）:
   ```
   <!-- OLD:START -->
   <span class="badge-item badge-date">📅2026年5月24日 08:12 JST</span>
   <!-- OLD:END -->
   ```

3. **セクション内で同一の old_str を複数回使用しない**
   - 同じ文字列が index.html に複数存在する場合は
     それぞれ異なる固有コンテキストを含めた old_str を使う

### news_data.json の更新（[S10]）

news_data.json の変更も **APPLY ブロック形式**で記載する。
ブロック内に `<!-- FILE:docs/data/news_data.json -->` を追加することで
updater.html が自動的に news_data.json を対象ファイルとして認識する。

#### フォーマット例

```markdown
## [S10] news_data.json 更新

<!-- APPLY:START -->
<!-- FILE:docs/data/news_data.json -->
<!-- OLD:START -->
  "updated": "2026年5月23日 09:58 日本時間JST",
<!-- OLD:END -->
<!-- NEW:START -->
  "updated": "2026年5月24日 08:12 日本時間JST",
<!-- NEW:END -->
<!-- APPLY:END -->

<!-- APPLY:START -->
<!-- FILE:docs/data/news_data.json -->
<!-- OLD:START -->
  "latest": [
<!-- OLD:END -->
<!-- NEW:START -->
  "latest": [
    {
      "title": "新規ニュースタイトル",
      "body": "本文...",
      "sourceLabel": "Reuters",
      "date": "2026年5月24日",
      "label": "🕊️ 外交",
      "url": "https://..."
    },
<!-- NEW:END -->
<!-- APPLY:END -->
```

#### ルール

- `<!-- FILE:docs/data/news_data.json -->` は `<!-- OLD:START -->` の直前に記載
- old_str は JSON の実際の文字列と**完全一致**させる（インデント・カンマも含む）
- latest 配列への先頭追加は `"latest": [` を old_str として新規アイテムを prepend
- staleNotice・updated の更新も個別の APPLY ブロックで記載
- FILE: 指定がない APPLY ブロックは従来どおり index.html に適用される

## ⛔ diffs.md 生成時の絶対ルール（CLAUDE.md 共通）

以下は Claude.ai が diffs.md を生成する際に**絶対に守らなければならないルール**。
APPLY ブロックの有無に関わらず常に適用される。

1. **コメントアウトタイトルを変更しない**
   - `<!-- xxxx -->` 形式のセクション識別コメントは変更・削除しない
   - 例：`<!-- SITUATION CARDS -->`・`<!-- COUNTDOWN -->` 等

2. **更新対象セクション以外を触らない**
   - diffs.md に記載したセクション（S01〜S11）以外の HTML は一切変更しない
   - 「ついでに修正」は厳禁。別の更新作業として明示的に依頼する

3. **タイムスタンプは必ずユーザーが明示した日時を使用する**
   - Claude が自動推測・補完した日付は使わない
   - ユーザーが「09:58 JSTで更新」と明示した場合はその時刻のみを使用する
   - 日付・時刻が明示されていない場合は**作業前にユーザーに確認する**

4. **ニュース URL は AI 生成・推測・捏造を絶対禁止**
   - web 検索で実在を確認済みの URL のみ使用可
   - URL が見つからない記事は latest に入れない（osint へ回すか別記事で代替）

## なぜこのルールが必要か

`diffs.md` を「記憶と慣習」で生成すると、セクションが漏れる。  
連日の更新漏れ（速報インシデント・シナリオ確率補足バナー等）の根本原因はここにある。  
以下のスケルトンを毎回の生成起点として使うことで漏れをゼロにする。

---

## Claude.ai の絶対手順

`diffs.md` を生成する際は、以下を必ず守ること：

0. **【必須・最初に実施】直前の diffs.md と更新ログを確認し「現在のHTML実態」を把握する**

   同日に複数回の更新が行われることがあるため、
   **直前の diffs.md の `<!-- NEW:START -->` 内容** が現在の HTML の実態である。
   これを確認せずに old_str を推測・仮定してはならない。

   #### 確認手順（必須）

   **Step A: 最新の diffs.md を取得**
project_knowledge_search: "index_html_diffs.md 最新 更新 JST"
   → 取得した diffs.md の各セクションの `<!-- NEW:START -->` 内容 = 現在の HTML に存在する文字列

   **Step B: 更新ログの最新行を確認**
project_knowledge_search: "更新ログ 出典 JST 更新"
   → `<!--出典・更新ログ-->` の先頭行（例：`2026年5月24日 13:45 JST 更新`）で
     「最後に適用された更新」の日時を把握する

   **Step C: 把握した日時から old_str を設定**
   → 直前の diffs.md の NEW:START 内容を old_str として使う
   → 1日に複数回更新がある場合、08:12 の次が 13:xx なら
     13:xx の NEW:START が現在の実態（08:12 の NEW:START ではない）

#### ⚠️ run.bat 未使用時の特別ルール（必須）

   run.bat（auto_push.py）を使用しなかった更新回では、
   diffs.md は GitHub に push されないため、
   プロジェクト知識内の diffs.md は**古い状態のまま**となる。

   - Step A（diffs.md 検索）の日時 ≠ Step B（更新ログ）の日時 の場合
     → **Step B を正として採用し、Step A の diffs.md は参照しない**
   - Step A と Step B の日時が一致する場合のみ、diffs.md の old_str を参考にしてよい
   - 判断に迷う場合は必ず Step B（更新ログ先頭行）を優先する

   **NGパターン（禁止）**：
   - diffs.md の日時と更新ログの日時が食い違っているのに、
     diffs.md 側の日時を「最新」と宣言する

   #### この手順が必要な理由

   - 1日に複数回 run.bat を実行した場合、HTML は更新のたびに状態が変わる
   - Claude.ai のプロジェクト知識検索は「キーワード一致スコア」で古い
     スニペットが上位にヒットすることがあり、最新状態を誤認しやすい
   - 「Sync Now 済み」でも検索クエリ次第で古い内容が返る
   - **この確認を省略すると old_str の大量ミスマッチが連鎖する**

   #### NG パターン（禁止）
   - 「前回の更新は08:12だったはず」という記憶・推測で old_str を書く
   - 最新 diffs.md を確認せずに直前セッションのメモリで判断する

1. 下記「必須セクション対応表」を確認する
2. 全11セクションのスケルトンを先に展開する（内容は空欄のまま）
3. 各スケルトンに情報を埋める
4. 全11セクションが埋まっていることを確認してからファイルを出力する
5. ダウンロードリンクを提示する前に下記「出力前セルフチェック」を実施する

## ⚠️ 特に漏れやすい2セクション（要注意）

### [S03] 速報インシデント が漏れる理由

- HTML の位置が MAP と STATS の後（ページ後半）にあるため
- TICKER や情勢カードより目立たない
- 「どうせニュースカラムと重複する」と思い込みやすい

→ **対策：必ず diffs.md の S03 スロットを先に作り、そこに書く**

### [S06] シナリオ確率補足バナー が漏れる理由

- CLAUDE.md に「シナリオ確率の数値は自動同期・手動更新不要」と書かれているため「補足バナーのテキストも不要」と誤解しやすい
- シナリオ4本（[S07]）と同じ `<!-- SCENARIOS -->` 内にあるため「[S07] を更新すれば [S06] も含まれる」と誤認しやすい

→ **対策：補足バナーは S06 として独立セクション扱いにし、必ず別途書く**

### 生成順序の制約（重要）

- **[S08.5] 全ルート現況サマリーは S08（シナリオフッター）完了後・S09（30秒カラム）の直前に書く**
  → ルート情報に変化がない週も日付を更新し末尾に「再確認済」を付ける。
  → 形式：変更あり → `YYYY年MM月DD日 HH:MM JST 更新` ／ 変更なし → `YYYY年MM月DD日 HH:MM JST 再確認済`

- **[S09] 30秒カラムは必ず全セクション確定後に最後に書く**
  → 30秒カラムは全セクションの総括。S01〜S08・S10・S11 が全て埋まってから着手する。
  → 先に書いた場合、後続セクションの情報更新と矛盾が生じる。

- **[S07] シナリオC（完全封鎖）とD（軍事エスカレーション）は必ず内容を差別化する**
  → 近似した文章になっていないか出力前に必ず読み比べる。
  → WTI価格レンジもシナリオ escalation 度に沿って変える（C と D で同一数値は NG）。

### 日付フォーマット統一

すべての日付・タイムスタンプは以下のフォーマットで統一する：

```
YYYY年MM月DD日 HH:MM 日本時間JST
例：2026年5月10日 12:00 日本時間JST
```

- **Claude が自動推測した日付は使わない**
- タイムスタンプの日付は必ずユーザーがプロンプトで明示したものを使用する
- 日付が明示されていない場合は作業前にユーザーに確認する
- TICKER 内の日付には必ず「JST」を付ける（例：`5/10 09:51 JST` ← OK　`5/10` のみ ← NG）

### ⚠️ [S10] latest 記事の「内容とURL一致」チェック（新設・必須）

**latest 配列の各アイテムは「title・body の内容」と「url・date・sourceLabel」が
同一記事を指していなければならない。**

#### 禁止パターン（今回の事故の再現）
❌ 悪い例：
title: "ルビオ国務長官「和平交渉に若干の好兆候」"  ← 5/21の発言
body: "ルビオは5月21日に…と発言"               ← 5/21の内容
url: "https://cnbc.com/2026/05/11/..."           ← 5/11の別記事
date: "2026/05/11"                               ← 不一致

#### 正しい手順（必ず守る）
✅ 正しい手順：

1. 書きたいイベント（例：ルビオ5/21発言）を決める
2. そのイベントを報じた承認済みソース（Bloomberg・Reuters・CNBC・NPR・AP等）のURLを検索する
3. URLが見つかった → そのURLのtitle・bodyを書く
4. URLが見つからない → その記事は latest に入れない（osintへ回すか、別イベントで代替する）
5. 「内容」と「URL」を別々に決めてから合体させることは禁止

#### セルフチェックに追加（S10確認項目）
- [ ] S10 latest ― 各記事のtitle/bodyは、そのurlが指す記事の内容を要約しているか
（urlと内容を別々に決めて後から合体させていないか）← 今回の事故防止

### ⚠️ osint 検索の必須化（変更なし判断の前に必ず実施）

**osint を「変更なし」にする前に、以下の検索を必ず実施すること：**
aljazeera.com iran war liveblog [YYYY]/[M]/[DD]

- 当日付のライブブログが存在した場合は osint として追加する
- 「前日に更新した」という理由で検索を省略しない
- 「膠着継続日・静かな日」と判断した場合も省略しない（前日に大ニュースがあった翌日ほど反応が多い）
- 検索を実施した上で「新規記事なし」と判断した場合のみ「変更なし」と記載する

### ⚠️ osint（現地メディア視点）の更新ルール ← 履歴消失バグ防止

**osint 配列は「蓄積型」であり、毎回1件に置き換えるのは禁止。**

#### 正しい更新方法

```json
"osint": [
  { "isLatest": true,  "date": "2026/05/11", ... },  ← 今回追加（先頭に追加）
  { "isLatest": false, "date": "2026/05/09", ... },  ← 前回記事（isLatest を false に変更）
  { "isLatest": false, "date": "2026/05/06", ... }   ← それ以前（そのまま残す）
]
```

**ルール詳細：**
- 新しい osint 記事は配列の **先頭に追加** する
- 既存の記事は `"isLatest": false` に変えて **削除しない**（履歴として蓄積）
- osint 配列を **丸ごと1件に置き換えることは絶対禁止**
- `isLatest: true` は最新の1件のみに付与する（複数付与は NG）
- 表示は JS の `LIMIT=3` で先頭3件を常時表示。4件目以降は「過去の記事を見る」で折り畳み

**なぜこのルールが必要か？**
- osint には `archive` 機能がない（`latest` と異なる）
- 過去記事は osint 配列内に残すことが唯一の履歴保持手段
- 1件置き換えを繰り返すと、現地メディア視点の全履歴が消える（実際に消失事故が発生）

#### [S10] に記載すべき内容（osint 更新時）

```
osint：配列先頭に新記事を追加、既存記事は isLatest: false に変更してそのまま残すこと。
       配列ごとの置き換えは禁止。
```

｜📋 MM/DD HH:MM JST 確認済——新たな進展なし（膠着継続）

### シナリオ確率について

- `sc-tag-A/B/C/D` の確率数値は `syncScenarioFromDashboard()` が自動同期するため **手動更新不要**
- ただし矢印（↑↓）・補足テキスト・[S06] バナーの根拠文は **毎回手動で情勢に合わせて更新する**
- シナリオ確率補足バナーの日付と、[S07] シナリオ本文の内容が矛盾していないか確認する

> **⚠️ 数値記載禁止ルール（Claude.ai・Claude Code 共通）**
> `manual-update.json` の確率数値は **Gemini AI が `hormuz-data-` リポジトリを毎日自動更新する**。
> Claude.ai は diffs.md に確率数値を提案・記載しない。
> Claude Code は `manual-update.json` に数値を書き込まない。

## run.bat（auto_push.py）が扱うファイル

run.bat が自動 push する対象は以下の **1ファイルのみ**：

- `docs/tools/index_html_diffs.md`

それ以外のファイル（例：diffs-generation-rules.md・CLAUDE.md・Memory.md・news_data.json 等）は
run.bat では push されない。
Claude.ai がこれ以外のファイルの修正をClaude Codeに依頼する場合は、
**Claude Codeのチャットに直接 str_replace 指示を貼る**こと。
